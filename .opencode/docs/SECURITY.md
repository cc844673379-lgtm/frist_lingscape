# 安全与治理 (Security & Governance)

> 本文件汇总需求文档中所有安全、治理、合规相关条款。
> 涵盖治理六项内容、治理面板规则、数据边界、安全验收。
> 内容事实见 [docs/CONTENT.md §6](./CONTENT.md#6-安全与治理六项内容)。

---

## 1. 安全与治理区块定位

安全与治理是首页 **第 8 个独立区块**（§7.8），**不得与交付方法合并**，必须作为独立区块呈现。

**目标**：回应技术/信息化负责人关注点——模型接入是否统一；权限、成本、审计、路由是否可控；能否与现有系统、知识库和流程连接。

---

## 2. 治理六项内容（必须体现）

| 序号 | 治理项 | 说明 |
| --- | --- | --- |
| 1 | 模型调用审计 | 所有模型调用可追溯，记录调用方、时间、模型、token 消耗 |
| 2 | 权限与角色控制 | 基于角色的访问控制，不同角色可调用不同模型和知识范围 |
| 3 | 成本统计与预算控制 | 按部门/项目/角色统计成本，支持预算阈值告警 |
| 4 | 数据边界与企业知识治理 | 知识库分级分类，敏感数据边界控制，防止越权引用 |
| 5 | 多模型接入与路由策略 | 统一网关接入多模型，按场景/成本/能力路由 |
| 6 | 可观测可追溯可持续优化 | 调用链可观测，效果可度量，持续优化模型选择和提示策略 |

---

## 3. 治理面板规则

安全与治理区可使用"治理面板"式静态 UI 片段呈现。

**必须遵守**：
- 必须像真实企业系统片段，而不是装饰图。
- 可展示：模型调用状态、成本预算、权限角色、审计记录、知识库引用、流程节点。
- 数据若非真实，必须使用中性字段，如"调用审计""成本趋势""权限策略"，**不写夸张数字**。
- 警示不使用红色大面积铺底。
- 状态点使用 `--accent`。

**治理面板样式**：
| 属性 | 值 |
| --- | --- |
| 背景 | `--bg-panel` |
| 文字 | `--ink` / `--ink-soft` |
| 边框 | `1px solid --line` |
| 状态点 | `--accent` |

## 4. 数据边界与合规

### 4.1 内容合规红线
- **禁止虚构**客户名称（如 Acme Corp 等占位符）、数据或合作关系。
- 无真实案例时，必须使用「典型交付成果清单」或「交付能力清单」替代。
- 页面不得出现「概念稿」「draft」等对外不适合文案。

### 4.2 静态官网边界
- 静态官网不应伪装成具备后台和真实表单能力。
- 联系咨询不得实现真实表单提交（见 [docs/ARCHITECTURE.md §3.4](./ARCHITECTURE.md#34-联系咨询静态方式53)）。
- 不依赖数据库、服务端接口、账号系统或后台管理。

### 4.3 第三方资源
- 如使用第三方字体或库，需要确认授权和加载稳定性。
- 所有图片、字体、脚本应本地化或使用稳定 CDN。
- 首屏关键内容不依赖远程接口。

## 5. 品牌与法律风险

- 品牌名称、英文名、域名、商标和社交平台名称必须单独核查，本需求不构成法律意见。
- 公司主体"湖北奥羚技术有限公司"必须在关于公司区明确出现。
